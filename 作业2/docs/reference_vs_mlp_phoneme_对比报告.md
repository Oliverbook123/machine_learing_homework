# reference_local.py vs mlp_phoneme.py 对比报告

## 一、概述

两份代码均实现音素分类（Phoneme Classification）的多分类 MLP 模型，使用 TIMIT 数据集（429 维 MFCC 特征 → 39 类音素）。以下是两个版本的全方位对比。

| 维度 | reference_local.py（参考代码） | mlp_phoneme.py（自实现） |
|------|-------------------------------|--------------------------|
| 运行环境 | CUDA（云端服务器） | CPU（本机 Mac） |
| 训练时长 | 40 epochs（云端 CUDA 较快） | 12 epochs（本机 CPU 较慢） |

> **注意：** 运行设备（CUDA vs CPU）不影响模型精度，仅影响训练速度。以下所有精度对比反映了模型架构和超参的差异，而非设备差异。

---

## 二、模型架构对比

### 2.1 核心架构

| 维度 | reference_local | mlp_phoneme |
|------|----------------|-------------|
| 代码组织风格 | 单独成员变量（`self.layer0`、`self.layer1`...） | `nn.Sequential` 容器 |
| 隐藏层层数 | 4 层 | 3 层 |
| 网络结构 | 429 → **2048** → **1024** → **512** → **128** → 39 | 429 → **512** → **256** → **128** → 39 |
| 总参数量 | **更大**（约 2.8M） | **更小**（约 0.5M） |
| BatchNorm | 每隐藏层后均有 | 每隐藏层后均有 |
| Dropout 概率 | 0.25 | 0.30 |

### 2.2 参数量估算

**reference_local**（4 个宽隐藏层）：
- layer0: 429×2048 + 2048 = 880,640
- layer1: 2048×1024 + 1024 = 2,098,176
- layer2: 1024×512 + 512 = 524,800
- layer3: 512×128 + 128 = 65,664
- out: 128×39 + 39 = 5,031
- **线性层合计：约 3,574,311 个参数**（加上 4 个 BN 的 γ/β：约 7,424 个）

**mlp_phoneme**（3 个窄隐藏层）：
- Linear1: 429×512 + 512 = 220,160
- Linear2: 512×256 + 256 = 131,328
- Linear3: 256×128 + 128 = 32,896
- out: 128×39 + 39 = 5,031
- **线性层合计：约 389,415 个参数**（加上 3 个 BN 的 γ/β：约 1,792 个）

**结论：** reference 模型的参数量约为 mlp_phoneme 的 **7 倍**。

## 三、训练配置对比

### 3.1 超参数总表

| 超参数 | reference_local | mlp_phoneme | 差异分析 |
|--------|----------------|-------------|----------|
| Epochs | **40** | **12** | reference 训练轮次更多 |
| Batch Size | **64** | **256** | mlp_phoneme 每批数据量更大 |
| 验证集比例 | **20%**（245,987 条） | **10%**（122,994 条） | mlp_phoneme 更多训练数据 |
| 优化器 | **NAdam** | **Adam** | NAdam 是 Adam + Nesterov 动量 |
| 初始学习率 | **0.0001** | **0.001（余弦退火衰减）** | mlp_phoneme 初始 LR 更大 |
| 权值衰减(L2) | 无 | 0.0001 | mlp_phoneme 含正则化 |
| 学习率调度 | 无（固定 LR） | CosineAnnealingLR | mlp_phoneme 有 LR 衰减 |
| Dropout | 0.25 | 0.30 | mlp_phoneme 丢弃更多 |
| 后处理平滑 | 无 | 窗口 size=7 均值滤波 | mlp_phoneme 额外步骤 |

### 3.2 区别要点解读

1. **验证集比例不同：** reference 用 80/20 划分，mlp_phoneme 用 90/10。mlp_phoneme 训练数据更多（+10%），但验证集更小，验证指标方差更大。

2. **Batch Size 差异：** reference 的 batch_size=64，梯度更新频繁但噪声大；mlp_phoneme 的 batch_size=256，梯度更稳定但占用显存更大。

3. **学习率策略：** reference 使用固定 LR=0.0001 + NAdam 优化器，收敛平稳但需要更多 epoch；mlp_phoneme 从 LR=0.001 开始通过余弦退火逐渐降为 0，配合 weight_decay 正则化，在更少 epoch 内达到较好的收敛效果。

4. **后处理：** 只有 mlp_phoneme 做了滑动窗口均值滤波（size=7），将孤立预测噪声平滑掉。约 31 万样本（占测试集的 70%）被后处理修改。

## 五、优劣分析

### 5.1 reference_local 的优势与代价

**优势：**
- 模型容量大（4 层宽 MLP，~3.58M 参数），拟合能力更强
- 更多 epoch（40），收敛更充分
- 验证准确率更高（0.7508 vs 0.7269）

**代价：**
- 参数量是 mlp_phoneme 的 ~9 倍，训练和推理更慢
- 需要更多 epoch 来收敛（峰值在第 24 epoch，而 mlp_phoneme 在第 11 epoch 左右就已接近收敛）
- 无后处理，预测中的孤立噪声未被平滑

### 5.2 mlp_phoneme 的优势与差距

**优势：**
- 模型轻量（~0.39M 参数），训练速度快
- 12 epoch 即达到接近性能高点的 0.7269（参考在第 12 epoch 时为 0.7468）
- 含后处理平滑，有助于提升测试集上的一致性
- 余弦退火 + weight_decay 正则化，在更少 epoch 内高效收敛

**差距：**
- 最佳验证准确率比 reference 低约 2.4 个百分点
- 可能原因：模型容量不足（3 层窄 MLP vs 4 层宽 MLP）

---

## 六、结论与后续优化建议

1. **性能差距的主要原因：** 模型容量（3.58M vs 0.39M 参数）而非训练策略。mlp_phoneme 在 12 epoch 内的收敛效率很高，但容量瓶颈限制了上限。

2. **若要提升 mlp_phoneme 到 reference 水平，可尝试：**
   - 增加隐藏层宽度（如 1024→512→256 或 2048→1024→512→128）
   - 增加 epoch 数并配合早停
   - 后处理窗口可改用**众数滤波**而非均值滤波（整数类别上众数更合理）
   - 可尝试更大的训练集比例（如 80/20 划分）

3. **若要做 Kaggle 强基线（Strong Baseline），应采用 reference 的架构**并在此基础上优化（如增加更多层、更宽维度、调整正则化策略等）。
